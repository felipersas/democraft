Democraft emits stable `DCxxxx` diagnostics during static validation and live capture. Human-readable output includes repair suggestions; `--json` exposes the same fields for tools and coding agents.

## How diagnostics surface

- **CLI:** `democraft validate <demo>` prints them to stderr. Add `--json` for machine-readable output.
- **Studio:** the staleness badge and capture progress surface validation failures inline.
- **Capture:** assertion failures (`assert.visible`, `assert.text`, `assert.url`) and unresolvable targets produce diagnostics that fail the capture.

## Code ranges

| Range | Category |
|-------|----------|
| `DC001`‚Äì`DC099` | Definition and author callback failures. |
| `DC100`‚Äì`DC199` | Targets, scenes, steps, durations, and renderers. |
| `DC200`‚Äì`DC299` | Browser execution and capture failures. |
| `DC300`‚Äì`DC399` | Audio definition and timing failures. |

## Reading a diagnostic

A diagnostic has:

- **`code`** ‚Äî the `DC0xx` identifier.
- **`message`** ‚Äî human-readable description.
- **`path`** ‚Äî semantic location in `demo.ts`.
- **`suggestion`** ‚Äî a concrete repair action.
- **`docsUrl`** ‚Äî stable documentation for the code.
- **`demoId`** / **`sceneId`** / **`stepId`** ‚Äî where in the demo the issue is (when applicable).
- **`targetId`** ‚Äî the target name involved (for locator/assertion issues).

## Common codes

- `DC001`: invalid structure or configuration.
- `DC002`: duplicate scene or step ID.
- `DC003`: the author `run` callback threw.
- `DC101`: unknown target reference.
- `DC102`: invalid duration.
- `DC103`: invalid scene or missing callback.
- `DC104`: invalid step.
- `DC105`: unknown visual renderer.
- `DC106`: invalid target or locator list.
- `DC107`: unknown custom visual reference.
- `DC108`: custom visual props are not JSON-serializable.
- `DC201`: browser execution, assertion, target resolution, or screenshot failure.
- `DC300`‚Äì`DC306`: duplicate audio IDs, missing sources, invalid volume,
  invalid spans or fades, unsupported extensions, and malformed durations. See
  the [audio guide](../guides/audio).

Authentication preflight failures use semantic `AUTH_*` error codes rather
than `DCxxxx`, because they describe a reusable session prerequisite rather
than the demo definition. See [authentication profiles](../guides/authentication)
for expiry, renewal, JSON output, and agent actions.

<Callout type="info">
The canonical codes and their producers live in `packages/schema/src/diagnostics.ts`, `packages/compiler/src/validation.ts`, and `packages/playwright/src/diagnostics.ts`. Once a code is released, it must not be repurposed for a different failure.
</Callout>

## For AI agents

When generating a `demo.ts`, run `democraft validate <demo>` and fix every diagnostic before attempting a capture. The static pass catches structural issues (unknown target names, malformed steps, bad durations) without needing the target app ‚Äî so it is cheap to iterate against.
