O Democraft emite diagnostics `DCxxxx` estáveis durante a validação estática e a captura ao vivo. A saída legível inclui sugestões de correção; `--json` expõe os mesmos campos para ferramentas e coding agents.

## Como os diagnósticos aparecem

- **CLI:** `democraft validate <demo> --static` os imprime no stderr. Adicione `--json` para saída legível por máquina.
- **Studio:** o badge de staleness e o progresso de captura mostram falhas de validação inline.
- **Captura:** falhas de asserção (`assert.visible`, `assert.text`, `assert.url`) e alvos não resolvidos produzem diagnósticos que falham a captura.

## Faixas de códigos

| Faixa | Categoria |
|-------|-----------|
| `DC001`–`DC099` | Definição e falhas no callback do autor. |
| `DC100`–`DC199` | Targets, cenas, steps, durações e renderers. |
| `DC200`–`DC299` | Execução no navegador e falhas de captura. |

## Lendo um diagnóstico

Um diagnóstico tem:

- **`code`** — o identificador `DC0xx`.
- **`message`** — descrição legível por humanos.
- **`path`** — localização semântica no `demo.ts`.
- **`suggestion`** — uma ação concreta de correção.
- **`docsUrl`** — documentação estável do código.
- **`demoId`** / **`sceneId`** / **`stepId`** — onde na demo está o problema (quando aplicável).
- **`targetId`** — o nome do target envolvido (para problemas de locator/asserção).

## Códigos comuns

- `DC001`: estrutura ou configuração inválida.
- `DC002`: ID duplicado de cena ou step.
- `DC003`: o callback `run` do autor lançou uma exceção.
- `DC101`: referência a target desconhecido.
- `DC102`: duração inválida.
- `DC103`: cena inválida ou callback ausente.
- `DC104`: step inválido.
- `DC105`: renderer visual desconhecido.
- `DC106`: target ou lista de locators inválida.
- `DC201`: falha no navegador, assertion, resolução de target ou screenshot.

<Callout type="info">
A lista completa de códigos vive em `packages/schema/src/diagnostics.ts`. Códigos são estáveis entre versões — um diagnóstico que emite `DC042` hoje emitirá `DC042` após um upgrade. Novos códigos são apenas adicionados, nunca reutilizados.
</Callout>

## Para agentes de IA

Ao gerar um `demo.ts`, rode `democraft validate <demo> --static` e corrija cada diagnóstico antes de tentar uma captura. O passe estático captura problemas estruturais (nomes de alvo desconhecidos, passos malformados, durações ruins) sem precisar do app-alvo — então é barato iterar contra ele.
